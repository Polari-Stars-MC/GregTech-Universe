## @Kogasa-Studio FABRICATIO

### 1. 总体方向

- 项目希望同时覆盖较完整的科技线与太空探索线。
- 科技线可包含矿物处理、化工、石油、核能等工业体系。
- 机器形态倾向于类似 HBM 的多方块单模型，并辅以少量纯方块构成的巨构。

### 2. 科技研究玩法

- 玩家可通过观察世界、消耗资源或时间来获取研究点。
- 研究过程不只是点树，而是分为“产生思路 -> 选择方向 -> 构建理论 -> 进行实验 -> 对照数据 -> 解锁节点”。
- 理论构建与实验验证可设计为独立玩法，例如小游戏、实验室设备、单方块或多方块实验平台。
- 科技解锁应体现“认知推进”而非单纯配方门槛。

### 3. 太空探索框架

- 起始舞台是一个偏拟真的太阳系。
- 设定中有生命与生态的行星不超过 4 颗，另有 3 到 5 个气态巨星。
- 轨道系统倾向二维化，而非完全复刻 KSP 式三维轨道。
- 入轨、转弯、降落中的部分复杂阶段可由系统自动处理，保留核心决策与机动选择。
- 玩家发射火箭后自动进入低轨圆轨道，再选择目标并进行转移机动。
- 抵达目标天体的捕获范围后，自动完成捕获与圆轨。

### 4. 太空空间表达

- 太空距离采用按比例缩放的坐标表达，例如数十万公里映射为数百格移动。
- 太空中不允许直接放置方块。
- 每个航天器对应一个独立子维度，建造行为仅在该子维度内进行。
- 通过对接接口可实现多个航天器子维度的合并。
- 小行星带、土星环等内容可以考虑做成独立维度或独立场景。

### 5. 科技线与太空线的关系

- 不打算对太空探索设置强制性的科技硬门槛。
- 玩家即使未完成完整科技研究，也可以依靠操作技术强行推进航天活动。
- 但环境适应技术会显著影响生存与任务成功率。
- 例如：
  - 未研究宇航服，仍可上天，但会迅速死亡。
  - 观测到火星存在大气后，才能合理研究隔热底。
  - 探测器返回更多数据后，才能进一步研究低温防护、重力适应、防冻设备等。
- 科技线更适合作为“降低风险、扩大能力边界”的系统，而非唯一通行证。

## 神舟：关于具体技术实现的一个点子

> 不确定实际可行性如何，偏向技术草案。

### 1. 维度划分

- 每个星球地表使用独立的 `planet dimension`，简称 `PD`。
- 进入太空后切换到独立的 `space dimension`，简称 `SD`。
- 在 `SD` 中，航天器以 `0 0 0` 为中心存在，本体不实际移动，移动与轨道变化主要由外部管理器驱动。
- 这样做的目标是尽可能兼容 BE，并规避机械动力与 Valkyrien 等方案带来的复杂度。

### 2. 轨道与外部渲染

- 所有星球和飞船的轨迹统一交给 `OrbitManager` 管理。
- 玩家从自身飞船内部向外观察时，看到的星球和其他飞船主要是渲染结果，而非真实加载实体。
- 飞船机动只更新轨道参数，再结合玩家自身旋转，生成舱外画面。

### 3. 对接方案

- 对接时，对方飞船也可以仅作为渲染对象显示。
- 对接完成后，通过过场加载，将质量较小的飞船并入质量较大的飞船所在维度。
- 这一方案的核心优点是实现简单、兼容性高。
- 明显缺点是不支持未对接状态下的 EVA 交互。

### 4. 着陆与起飞切换

- 着陆阶段看到的地面也可主要通过渲染呈现。
- 客户端需要同时加载 `PD` 与 `SD`：
  - 主摄像机位于 `SD`
  - 后台摄像机位于 `PD`
- 着陆完成后，通过过场将飞船实体加载到 `PD`，再把前台切换到 `PD` 并销毁 `SD`。
- 起飞流程则反向执行。

### 5. IVA / EVA 与重力

- 玩家在 IVA 或无人工重力的 EVA 状态下，可以进行三轴自由旋转，类似幻翼飞行。
- 此时航天器本体仍可保持静止，运动效果主要通过渲染与轨道参数体现。
- 可进一步考虑支持任意方向的人造重力，但实现难度较高，更适合作为远期目标。

### 6. 星球生成

- 除主世界和月球外，其余星球可采用程序化生成。
- 程序化内容既包括世界生成，也包括远距离观察所需的渲染模型生成。
- 星球形态上，立方体可能是比球形或“地平说”更现实的折中。
- 大气视觉效果可以等 VV 更新后再补，以降低对光影方案的依赖。

## @baka4n（思念）

### 1. 动态资源生成

- 打破常规地形生成机制。
- 星球资源可在玩家初次进入时，依据该星球的资源配比动态生成。
- 每个区块拥有一定浮动比例，但整体应逐步逼近星球资源设定的总量平衡。

### 2. 宇宙空间表达

- 宇宙设定不应被简单理解为“另一个维度”。
- 传统维度体系更像是高度受限的空间容器，不适合作为完整宇宙表达方式。
- 宇宙位置、玩家位置，以及它们之间的渲染关系，应基于星系内相对位置来处理。

### 3. 飞船存储结构

- 飞船不应继续依赖“维度式存储”。
- 更适合建立项目自定义的空间与存档结构，用于保存船体方块数据。
- 该结构可被定义为“船体结构”，作为独立于世界维度之外的核心数据单位。

## 辉夜：整理一堆想法

### 1. 底层存档与坐标系统

- 重构原版存档系统。
- 使用 `double` 存储实体位置，使用 `long` 存储 `xyz` 坐标。
- 重定义空间分块方式，例如：
  - `chunk = 16^3`
  - `region = 256^3`
- 目标是支持更大尺度的世界、程序化地形与地质生成，以及更深层的地下探索。
- 代价是需要重写大量底层内容。

### 2. 光照与渲染

- 禁用原版光照引擎。
- 光照渲染改由 shader 方案承担。
- 光源应区分恒星与人造光源两类。
- 目标是同时解决性能、表现力和可控性问题。

### 3. 生存压力设计

- 玩家在探索中的主要威胁应来自环境，而非传统怪物堆叠。
- 非宜居星球的核心环境压力包括气压、温度、辐射等。
- 宜居星球也可以存在气体成分异常、毒性环境或危险生物群落。

### 4. 战斗与武器控制

- 可能需要单独编写火控系统来控制武器。
- 该部分目前仍属于开放议题，需视整体玩法定位继续细化。
